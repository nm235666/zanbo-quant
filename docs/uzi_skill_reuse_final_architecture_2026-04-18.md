# UZI-Skill 可复用实现终局蓝图（2026-04-18）

## 1. 文档目的

本文件定义“可直接复用到 Zanbo Quant 的实现能力终局态”，目标不是最小可用，而是一次定义清楚最终形态，便于后续按批次拆分执行并持续验收。

适用范围：
- 前端展示业务主链（`stocks/signals/research/system`）
- 决策闭环与证据体系
- 回归门禁与质量治理

不包含：
- 后端协议破坏式改造
- 与主链无关的实验功能

---

## 2. 终局目标（Final State）

系统达到以下状态才算“复用落地完成”：

1. 决策流程有统一入口，不同角色不会在入口层分叉成孤岛流程。  
2. 任意关键结论都有可追溯证据（来源、时间、版本、快照）。  
3. 输出发布前有硬门禁，不满足质量规则时不可发布。  
4. 已知问题会沉淀为长期检查规则（BUG->CHECK），避免重复踩坑。  
5. 同一分析任务可按深度策略稳定复现（quick/standard/deep）。  
6. 会话/页面启动时自动注入上下文，不依赖用户反复手填背景。  
7. 弱数据或源异常时有可信 fallback，不出现“空白可用假象”。  
8. 产出同时支持机器消费（JSON）与人工决策（Markdown）双轨输出。  

---

## 3. 可复用能力与最终实现效果

## 3.1 机械硬门禁（Hard Gate）

最终效果：
- `research/decision`、报告导出、策略建议发布前，统一经过 `QualityGate`。
- 若触发关键阻断项（证据不足、结论不可追溯、冲突未解释、时效超限），前端禁止“发布/提交”。
- 用户看到明确阻断原因、修复建议、责任字段，不是抽象失败码。

实现要点：
- 统一阻断等级：`blocker / warning / info`。
- `blocker` 必须阻止提交；`warning` 允许提交但需二次确认并记录原因。
- 每次 gate 结果可审计（谁、何时、哪条规则、是否豁免）。

验收标准：
- 高风险路径 100% 接入 gate。
- blocker 漏拦截率为 0。
- 任一发布动作可回查 gate 记录。

## 3.2 BUG->CHECK 规则沉淀

最终效果：
- 每个线上事故或高频缺陷，必须转成结构化检查项并纳入门禁。
- 规则库可按模块、严重度、生效范围管理，支持版本化。
- 回归执行时自动跑规则，不再靠“记得手测”。

实现要点：
- 规则实体至少包含：`rule_id`、`source_bug`、`scope`、`predicate`、`severity`、`owner`、`created_at`。
- 规则命中后返回可行动信息（复现线索 + 修复方向）。

验收标准：
- P0/P1 缺陷转规则覆盖率 >= 95%。
- 重复缺陷率连续 3 个迭代下降。

## 3.3 深度策略档位（Analysis Profile）

最终效果：
- 分析任务标准化支持 `quick/standard/deep` 三档。
- 三档在抓取范围、推理预算、证据阈值、输出结构上明确可区分。
- 同一输入在同一档位可稳定复现，不因调用入口变化而漂移。

实现要点：
- 深度策略由统一配置中心提供，禁止页面私配参数。
- 每次分析结果回显“本次档位 + 生效参数摘要”。

验收标准：
- 三档行为差异可量化、可解释。
- 同档位复跑结果波动在可控范围内。

## 3.4 命令化阶段协议（Protocolized Workflow）

最终效果：
- 关键分析链路统一为阶段协议：`输入 -> 取证 -> 归因 -> 判断 -> 动作建议 -> 风险披露 -> 输出`。
- 每个阶段有状态、耗时、输入输出摘要，可观测可中断可恢复。
- 不允许“直接跳到结论”绕过取证与风险披露阶段。

实现要点：
- 后端任务状态机与前端阶段条一一对应。
- 中断后重进页面可恢复到上次阶段与上下文。

验收标准：
- 主链路阶段可视化覆盖率 100%。
- 中断恢复成功率 >= 99%。

## 3.5 启动上下文注入（Session Context Injection）

最终效果：
- 从 `stocks/scores`、`signals/graph`、`news` 等入口进入 `research/decision` 时，上下文自动注入且显式展示。
- 上下文包含来源页面、筛选条件、时间窗口、触发意图。
- 用户可一键清空上下文，避免被隐藏过滤“锁住”。

实现要点：
- URL 与 store 双通道保存上下文，支持刷新与分享。
- 上下文作用域明确标注“仅影响默认查询”。

验收标准：
- 跨页进入决策板的上下文丢失率 < 1%。
- 用户对“当前为什么看到这些结果”可直接解释。

## 3.6 动态 fallback 证据兜底

最终效果：
- 当主数据源缺失、延迟或质量不足时，系统自动切换到可信候补源并标注来源等级。
- 弱数据状态下仍能给出“受限但可用”的输出，不出现空白画布或误导性确定结论。

实现要点：
- fallback 必须有可见标识：`primary / fallback / mixed`。
- 降级输出必须附带“置信度下降原因”。

验收标准：
- 弱数据场景可用性达标（页面可解释、可继续操作）。
- fallback 与主源混用场景下无来源混淆。

## 3.7 双轨证据产物（JSON + Markdown）

最终效果：
- 每次关键分析都生成：
  - 机器可消费的结构化证据（JSON）
  - 人工可审阅的决策摘要（Markdown）
- 两者共享同一 `trace_id`，可互相跳转。

实现要点：
- JSON 用于自动化校验、回归比较、指标计算。
- Markdown 用于投研协作、复盘、审阅留痕。

验收标准：
- 关键决策事件双轨产物覆盖率 100%。
- 任一结论可在 30 秒内定位到对应证据链。

## 3.8 回归门禁硬化（视觉 + 逻辑双门禁）

最终效果：
- 不仅跑功能 smoke，还必须跑关键页面截图回归与规则回归。
- 对 `stocks/signals/research/system` 设高风险门禁，失败即阻断集成。
- 审美与可读性指标进入准入标准（焦点、层级、稳定布局、状态清晰）。

实现要点：
- 视觉基线和逻辑基线统一归档，支持版本对比。
- “已知 flaky 用例”必须有明确 owner 和截止时间，不能永久豁免。

验收标准：
- 高风险路径回归阻断生效率 100%。
- 回归结果可追溯到页面、规则、责任人。

## 4. 与当前项目代码落点映射（终局责任分配）

后端与服务层：
- `services/decision_service/service.py`：承载阶段协议、质量门禁、证据装配总线。
- `backend/routes/decision.py`：暴露阶段状态、gate 结果、trace_id、上下文读写接口。
- `jobs/collect_daily_metrics.py`：沉淀长期证据指标、规则命中趋势、回归稳定性趋势。

前端页面与交互：
- `apps/web/src/pages/research/DecisionBoardPage.vue`：统一入口承接、上下文条、阶段可视化、双轨输出入口。
- `apps/web/src/pages/signals/SignalChainGraphPage.vue`：图谱入口到决策入口的上下文携带、弱数据兜底展示。
- `apps/web/src/pages/system/SourceMonitorPage.vue`：主源/fallback 状态、数据健康解释、降级可见化。

公共层：
- `apps/web/src/styles.css`：门禁反馈、状态等级、证据标签的统一视觉 token。
- `apps/web/src/stores/*`：上下文与阶段状态统一管理，禁止页面私有分叉状态。

测试与回归：
- `tests/` + `apps/web` e2e：规则回归、跨页上下文、阶段状态机、截图回归与阻断。

## 5. 终局验收口径（Definition of Done）

必须同时满足：

1. 主链路闭环成立：输入、判断、动作、跟踪、复盘全部可见可追溯。  
2. 关键结论可证据化：每条结论可回溯来源、时间、版本与 gate 记录。  
3. 跨模块联动成立：`stocks/signals/news/chatrooms -> research/decision` 上下文连通稳定。  
4. 异常与弱数据可解释：用户知道“发生了什么、为什么、下一步做什么”。  
5. 回归可阻断：高风险回归失败时无法进入下一交付环节。  

---

## 6. 非目标与边界

- 不以“技术重构完整度”替代“业务闭环完整度”。
- 不为了短期通过率关闭关键门禁。
- 不允许新增能力绕开统一协议、统一上下文、统一证据链。
