# Zanbo Quant 前端页面优化诊断与整改路线图

> 审计基准日期：`2026-04-04 UTC`
>
> 本文档基于当前 `apps/web/` 真实路由、共享 UI 组件、全局样式和代表性页面的静态审计结果整理，目标是为前端展示业务提供一份可直接排期的诊断清单与整改路线图。

## 2. 诊断清单

### 2.1 全局问题

| 方面 | 问题 | 原因 | 具体建议 | 优先级 |
| --- | --- | --- | --- | --- |
| 导航 / 响应式 | `xl` 以下缺少真实导航方案 | `AppShell` 侧边栏在 `xl` 以下直接隐藏，当前只有“展开/收起导航”按钮，没有移动端抽屉或底部导航承接 | 在 `AppShell` 上补一套 `xl` 以下可用的导航方案，优先使用抽屉式导航；保留当前导航分组与权限过滤逻辑，不做信息架构重构 | P0 |
| 表格 / 响应式 | 表格在小屏只有横向滚动，没有卡片替代视图 | `DataTable` 当前只提供 `overflow-auto`，对股票、信号、群聊等重表格页面，小屏可读性不足 | 为 `DataTable` 或表格页补移动端卡片模式，至少覆盖股票列表、信号总览、群聊总览、价格中心等主路径页面 | P0 |
| 表单 / 可访问性 | 大量筛选表单依赖 placeholder，缺少稳定 label | 多数页面筛选区采用“输入框矩阵”布局，字段名只靠 placeholder 表达，屏幕阅读器和快速扫描体验都较弱 | 为筛选区新增显式字段标签；placeholder 仅保留格式示例；优先覆盖股票、新闻、信号、群聊、研究页主筛选 | P0 |
| 可访问性 | 全站几乎没有基础 ARIA、表格语义增强和状态播报 | 当前代码中几乎没有 `aria-*`、`caption`、`scope`、`aria-live` 等基础语义补充 | 优先补基础语义：表格 `caption/scope`、状态更新 `aria-live`、按钮 `aria-label`、抽屉与可折叠区的展开状态声明 | P0 |
| 交互反馈 | 长任务页反馈偏文本化，进度与状态可视化不足 | 多角色分析、股票详情里的异步动作主要通过文案变化反馈，缺少阶段感和完成度感知 | 为长任务统一补“阶段状态 + 当前进度 + 最近更新时间 + 可重试/可取消状态”，优先覆盖多角色分析页和股票详情触发分析动作 | P0 |
| 布局 | 结果页首屏露出不足，筛选区和统计区占比偏高 | 多数结果页先展示完整筛选器和统计卡，真正的结果列表被推到首屏下方 | 对结果页采用“紧凑筛选条 + 可折叠统计区”策略，保证用户进入页面后更快看到结果数据 | P1 |
| 视觉层级 | 卡片、区块、渐变、圆角的视觉层级过于平均 | `PageSection`、`InfoCard`、`StatCard`、`MetricGrid` 风格相近，信息主次不够清晰 | 建立页面级、区块级、卡片级三层视觉层次，减少非关键卡片装饰，把视觉预算集中到首屏核心信息 | P1 |
| 壳层信息密度 | 页头区域承载标题、实时状态、最近事件、导航控制和退出动作，窄屏压力大 | `AppShell` 页头在桌面端信息完整，但对小屏和中屏没有做明显降噪 | 小屏时压缩页头信息，最近事件改为摘要或隐藏；把次要动作收进菜单，保留标题和状态主信息 | P1 |
| 状态表达 | 异常、警告、成功状态的对比度和形态差异不够强 | `StatusBadge` 和浅色容器普遍较接近，用户主要依赖文字判读状态 | 提高 warning/danger 的视觉区分度，并在关键错误场景增加图标、色条或独立异常容器 | P1 |
| 空态 / 错态 | 空态与错态大多停留在“暂无数据”或简单文案，缺少下一步动作 | 共享表格和多页面空态都偏静态，不利于引导用户恢复主流程 | 统一空态规范，区分“未查询 / 无结果 / 加载失败 / 无权限”，并在每类状态中补恢复动作 | P1 |
| 性能 | 长列表和长内容页存在重渲染风险 | 新闻、报告、群聊、日志、Markdown 页面普遍直接渲染全部内容，复杂页面同屏信息量大 | 对长列表引入分页后继续保留“渐进渲染”思路；对日志、Markdown、次级内容增加折叠、懒挂载或视口内加载 | P1 |
| 图表体验 | 图表虽有异步加载，但占位与加载感较弱 | 股票详情、价格中心等页面有多图表并存，当前占位多为简单文本框 | 为主路径图表补 skeleton 或结构化占位，优先提升股票详情与价格中心的首屏加载感 | P1 |
| 轮询策略 | 多页面固定轮询，但没有明确前台/后台节流策略 | dashboard、source monitor 等页面持续轮询，页面切到后台时仍可能保持同频率更新 | 为实时轮询页面增加失焦降频或手动刷新模式，先保证主展示页面不因后台轮询影响稳定性 | P2 |
| 设计系统 | 圆角、阴影、按钮样式在多个组件间接近但未形成明确规范 | 当前已有统一视觉方向，但没有明确的等级映射规则，后续扩展容易继续堆样式 | 以现有样式变量为基础补一份轻量设计约束：容器等级、状态色、按钮分层、数据卡规范 | P2 |

## 3. 整改路线图

### 第一阶段：前端主链路可用性修复

目标：先解决影响页面可达性、主流程可用性和基础无障碍的问题，优先保证前端展示主链路稳定可用。

任务包：

- 为 `AppShell` 补 `xl` 以下可用导航
- 为核心表格页补移动端替代视图或可读方案
- 为主筛选区补显式 label
- 为全站关键状态补基础 ARIA / 表格语义 / 状态播报
- 为多角色分析页和股票详情页补更明确的任务状态表达

完成定义：

- 小屏能进入主要业务页并完成导航切换
- 关键结果页在小屏下可以读懂主要数据
- 主路径筛选表单不再完全依赖 placeholder
- 关键异步动作能看出进行中、成功、失败三种状态

对前端展示业务的直接帮助：

- 先保证页面“能到、能看、能用”，减少导航断层和主流程阻塞。

对测试可用性的验证方式：

- 运行前端构建检查，并对导航、表格页、长任务页做手工 smoke 验证。

### 第二阶段：结果页体验与复杂页面信息层级优化

目标：提升结果页首屏露出、复杂页面的信息优先级和阅读效率，让页面更像研究终端而不是后台堆叠表单。

任务包：

- 压缩股票、信号、新闻、群聊等结果页筛选区占用
- 优化 `/dashboard`、`/stocks/detail`、`/stocks/prices`、`/signals/overview` 的首屏布局
- 为新闻卡、股票详情、群聊结果增加摘要层和展开层
- 调整全局容器层级，拉开页面级、区块级、卡片级差异

完成定义：

- 结果页进入后更快看到真实结果而非先看完整筛选墙
- 股票详情和多角色页面的主要结论在首屏更容易识别
- 首页研究信息与系统监控信息分区更清晰

对前端展示业务的直接帮助：

- 提升信息表达效率，让研究判断路径更短、更清楚。

对测试可用性的验证方式：

- 运行前端构建检查，并对首页、股票详情、信号总览、新闻页块做主流程对照验证。

### 第三阶段：性能与可访问性系统化收口

目标：在主链路稳定后，系统化补齐性能、语义和交互反馈规范，减少后续页面继续累积展示债务。

任务包：

- 为长列表、长 Markdown、日志区补渐进渲染和按需展开策略
- 为图表区补结构化 loading 和更稳定的挂载策略
- 为轮询页面补后台降频或刷新策略
- 形成轻量设计约束与页面验收标准，约束后续新增页面

完成定义：

- 长页面渲染压力下降，复杂内容默认不过度铺开
- 图表、状态、空态、错态具备统一表达方式
- 新增页面可按同一套检查标准验收

对前端展示业务的直接帮助：

- 让现有页面更稳、更清晰，并降低后续新增功能对展示质量的破坏概率。

对测试可用性的验证方式：

- 运行前端构建检查，并对关键长页面做性能观察和基础键盘可达性回归。

## 4. 优先级与验收规则

### 4.1 优先级解释

- `P0`：影响导航、可访问性、主流程可用性，优先保障
- `P1`：明显影响页面表达效率、结果可读性或操作反馈，应尽快安排
- `P2`：增强项或 polish，在主链路稳定后推进

### 4.2 统一验收检查项

凡是前端页面优化任务，默认至少检查以下项目：

- 页面是否能展示
- 数据是否能正常加载
- 空态 / 报错态是否有明确动作
- 小屏是否可用
- 关键操作是否有反馈
- 基础键盘可达性是否成立

### 4.3 文档使用方式

- 本文档优先用于前端展示业务排期，不直接替代详细开发设计稿
- 若后续实施中发现范围扩大，应拆成独立任务，不在一次改动中跨模块重构
- 新增页面或共享组件发生明显变化后，应同步更新本清单，保持“代码与文档一致”
