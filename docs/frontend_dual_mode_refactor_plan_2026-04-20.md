# 前端双模式重构执行方案

## 1. 文档目的

当前前端仍以“统一页面体系”组织，研究用户与后台治理能力混在同一产品表面，已经不利于主链收口、默认入口清晰化与后续长期维护。

本文档定义从“统一状态”重构为“用户模式 + 后台管理模式”的执行方案。它不是阶段性提案，而是后续导航、壳层、路由、权限和验收的唯一重构基线。

## 2. 重构目标

- 把研究用户从系统治理噪音中隔离出来。
- 把后台管理功能从研究主链中独立出来。
- 保证“用户模式”成为投资者默认工作入口。
- 保证“后台管理模式”成为管理员独立治理入口。
- 通过结构性分离而不是文案修补完成主链收口。

## 3. 最终前端结构

- `/app/*`：用户模式
- `/admin/*`：后台管理模式
- `/login`、`/upgrade`、`/403`、`/404`、认证回调页与通用错误状态页归入`公共共享页层`
- 公共共享页层不属于用户模式或后台管理模式，只负责登录、鉴权回调、权限提示、错误兜底与状态承接
- `/` 根据角色和权限自动分流：
  - 研究用户默认进入 `/app/workbench`
  - 管理员默认进入 `/admin/dashboard`
- 原有旧路径作为兼容层，逐步重定向到新路径

## 4. 模式职责定义

### 4.1 用户模式

用户模式服务于研究、判断、决策、执行、复盘，核心对象是“今天该看什么、该怎么做、后续如何跟踪”。

长期职责包括：

- 今日工作台
- 市场结论
- 候选漏斗
- 股票对象页
- 决策板
- 执行单
- 持仓
- 复盘
- 宏观三周期
- 长线配置

### 4.2 后台管理模式

后台管理模式服务于权限、任务、配置、监控、审计，核心对象是“系统是否稳定、规则是否正确、治理动作是否可执行”。

长期职责包括：

- 总控台
- 数据源监控
- 任务调度
- LLM 节点管理
- 角色权限
- 数据库审计
- 用户与邀请码管理

### 4.3 公共共享页层

公共共享页层不承担投资主链或治理主链职责，只负责跨模式公共入口与兜底状态：

- 登录
- 升级
- 权限错误
- 路由未命中
- 回调与通用状态页

## 5. 页面归属规则

- 任何直接服务投资任务完成的页面归用户模式。
- 任何直接服务系统治理、配置、监控、审计的页面归后台管理模式。
- 公共登录、鉴权、错误、回调类页面归公共共享页层。
- `stocks/*`、`signals/*`、`chatrooms/*`、`intelligence/*` 默认归用户模式的输入/分析层。
- 任一长期存在的页面必须有唯一主归属，不允许长期处于“双模式都像、双模式都挂”的状态。

边界页判定原则：

- 核心任务是帮助研究用户完成投资判断或动作，归用户模式。
- 核心任务是配置、监控、运维、权限、审计，归后台管理模式。
- 若同一页面同时承担两类主职责，默认判定为结构错误，应拆页而不是双挂。

## 8. 边界页案例

### 8.1 `stocks/*`

- 若页面主职责仍是研究对象查看，数据质量、采集状态、补数、审计信息只能作为附属诊断区，不改变主归属。
- 若治理动作成为主任务，应拆到后台管理模式独立页，而不是继续叠加在股票对象页中。

### 8.2 `chatrooms/*` 与 `intelligence/*`

- 面向研究用户的候选、情绪、事件、资讯消费页归用户模式。
- 若存在抓取状态、源配置、清洗规则、失败重跑、数据质量治理页，应拆到后台管理模式。

### 8.3 `signals/*`

- 面向研究判断的总览、主题、图谱、时间线归用户模式。
- 质量配置、审计、状态治理若长期以规则治理为主，应迁入后台管理模式或拆出 admin 子页。

### 8.4 `system/users`

- 研究身份信息展示如果只是辅助查看，不改变页面归属。
- 只要页面主任务是用户管理、权限分配、会话控制、邀请码、审计操作，就归后台管理模式。

### 8.5 LLM 失败页或任务失败详情

- 若主目标是帮助研究用户理解本次分析失败并继续任务，归用户模式。
- 若主目标是排查节点、限流、provider、任务堆积与治理原因，归后台管理模式。

## 9. 壳层重构要求

- 新增两个清晰壳层：
  - `UserShell`
  - `AdminShell`
- 两套壳层必须在以下方面明显区分：
  - 导航结构
  - 默认落点
  - 信息密度
  - CTA 风格
  - 面包屑与任务语义
- 用户模式以“今日任务、结论、动作”为中心。
- 后台管理模式以“配置、状态、监控、告警、审计”为中心。

## 10. 导航重排原则

- 用户模式导航只保留投资主链所需内容：
  - 工作台
  - 市场
  - 候选
  - 决策
  - 执行
  - 持仓
  - 复盘
  - 宏观
  - 配置动作
- 后台管理模式导航只保留治理内容：
  - 总控台
  - 数据源
  - 任务
  - LLM
  - 权限
  - 审计
  - 用户管理
- 公共共享页层不承担主导航职责。
- 禁止在用户模式主导航中混入系统治理、监控、审计、配置类入口。
- 禁止后台管理模式承担研究用户默认工作入口职责。

## 11. 权限、分流与模式切换规则

- 研究角色默认进入用户模式。
- 管理角色默认进入后台管理模式。
- 兼具研究与管理权限的账号必须支持双向切换。
- 模式切换是显式能力，不依赖手动改 URL。
- 在 `/admin/*` 下应能一键进入 `/app/workbench`。
- 在 `/app/*` 下应能一键进入 `/admin/dashboard`。
- 切换后保持账号身份、权限与会话不变，只切换使用面、导航与任务语义。
- 模式切换必须默认保持：
  - 登录身份不变
  - 权限上下文不变
  - 会话状态不丢失
  - 能保留的对象上下文尽量保留

## 12. 对象上下文保留规则

### 12.1 保留粒度

- 优先保留 `stock ts_code`
- 其次保留 `theme / signal / industry / query context`
- 再其次保留筛选条件、时间窗口、排序与分页
- 不保留仅在当前模式内部有意义的治理态临时参数

### 12.2 允许跨模式映射的对象

- 股票对象
- 主题对象
- 信号对象
- 研究上下文 query
- 可共享的时间区间与筛选条件

### 12.3 不允许直接跨模式映射的对象

- 后台内部任务运行态
- provider 节点配置态
- 审计与治理内部过滤条件
- 仅管理台存在的内部对象 ID

### 12.4 无对应页面时的降级规则

- 若目标模式存在对应对象页，则直接落到对象页。
- 若目标模式无对应对象页，但存在相关列表页，则带上下文落到列表页。
- 若无可承接页面，则跳转到目标模式默认首页。

### 12.5 切换反馈规则

- 模式切换后必须有明确提示。
- 推荐采用 `toast` 或页面顶部 `banner`。
- 提示至少包含：
  - 已切换到哪种模式
  - 当前保留了哪些上下文
  - 若有上下文丢失，需说明原因

## 13. 兼容迁移策略

- 第一阶段保留旧路径，对应跳转到新路径。
- 第二阶段统一更新导航、CTA、深链。
- 第三阶段清理旧路由直接入口，仅保留必要兼容跳转。
- 旧路径兼容期内，必须优先保障外部深链、测试脚本、收藏链接和站内 CTA 跳转不失效。
- 兼容目标不是“路径还能开”，而是“访问语义保持不变”。
- 禁止新增功能继续直接落入旧路径体系，除兼容跳转外不再扩展旧路径。
- 所有跨页链接和工作流跳转必须优先使用新路径。

## 14. 禁止事项

- 禁止在用户模式主导航中混入系统治理、监控、审计、配置类入口。
- 禁止后台管理模式承担研究用户默认工作入口职责。
- 禁止同一长期存在的页面同时以用户模式页面和后台管理页面双重身份存在。
- 禁止新增功能继续直接落入旧路径体系，除兼容跳转外不再扩展旧路径。
- 禁止通过文案弱化、导航隐藏或权限遮罩替代结构拆分。
- 禁止用户模式主链任务依赖用户手动跳转到 `/admin/*` 才能完成。
- 禁止后台治理任务依赖管理员手动跳转到 `/app/*` 才能完成。

## 15. 发布门禁

以下任一条件未满足，不得判定双模式重构达标：

- 研究角色访问 `/` 后不能默认进入 `/app/workbench`
- 管理角色访问 `/` 后不能默认进入 `/admin/dashboard`
- 用户模式主链仍存在必须进入后台管理模式才能完成的关键任务
- 长期页面仍存在无法归属到用户模式、后台管理模式、公共共享页层三者之一的情况
- 模式切换仍依赖手动改 URL，而非显式可操作入口
- 旧路径跳转后访问语义发生变化，导致深链、收藏或测试脚本失效
- 用户模式与后台管理模式权限解释不一致，或 upgrade 页未说明当前受限的是哪一种模式页面及可替代路径

## 16. 迁移完成定义

当以下条件同时满足时，才可判定双模式迁移完成：

- 所有长期页面均完成唯一归属
- 用户主链默认落在 `/app/*`
- 治理主链默认落在 `/admin/*`
- 公共共享页层范围稳定且不混入业务主任务
- 旧路径仅保留兼容跳转，不再承接新增功能
- 站内主 CTA、外部深链与测试脚本均已切换到新语义体系
- 用户模式与后台管理模式之间不存在关键任务互相借道完成的情况
