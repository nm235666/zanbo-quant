## 二、核心业务链路识别

### 1) 研究入口链路：登录 -> workbench/scoreboard -> decision

- 业务目标：快速进入当日研究与决策任务。
- 用户/平台意义：降低迷路成本，提高研究动作转化。
- 关键原因：决定用户是否形成稳定使用主线。
- 验证重点：跨入口是否稳定导向决策板并保留上下文。

### 2) 决策动作闭环链路：decision action -> execution task -> orders -> review

- 业务目标：把建议变为可执行、可追踪、可复盘。
- 用户/平台意义：决定系统是否真正支撑业务闭环。
- 关键原因：是“辅助决策工具”与“可运行决策系统”的分水岭。
- 验证重点：自动承接、状态回写、跨页追踪一致性。

### 3) 候选漏斗链路：候选 -> 信号 -> 决策

- 业务目标：将候选筛选对象化并可解释。
- 用户/平台意义：减少列表漂移，提升筛选效率。
- 关键原因：决定前置筛选质量与可控性。
- 验证重点：空态、流转、可追溯解释是否成立。

### 4) 权限门禁链路：受限访问 -> upgrade 解释 -> 替代动作

- 业务目标：避免“拒绝但无解释”。
- 用户/平台意义：降低低权限用户挫败感，提升可理解性。
- 关键原因：影响权限体系下的可用性与转化。
- 验证重点：多受限路由的一致性与恢复路径完整度。

### 5) 快速分析链路：对象页 -> quick insight -> 深度分析

- 业务目标：先快筛后深挖。
- 用户/平台意义：提升批量分析效率。
- 关键原因：决定 AI 能力是否“顺手可用”。
- 验证重点：入口可发现性、边界与降级语义。

## 三、详细问题清单

### 【问题标题】决策动作可重复提交，缺少稳定幂等保护

- 问题级别：高
- 问题类型：状态管理 / 业务闭环 / 异常处理
- 发现路径：`/login(pro) -> /research/decision -> 填写动作 -> 连续点击“确认”两次 -> 观察响应`
- 触发方式：重复操作
- 表面现象：同一操作意图生成两条不同 action（`action-7`、`action-8`），未出现 dedup。
- 深层问题判断：前端提交锁与后端幂等协同不足，关键写操作可被双击放大。
- 影响分析：
  - 用户理解成本：中
  - 任务完成率/转化率：中（重复动作污染链路）
  - 操作效率：中低（后续清理成本）
  - 信任感：高影响
  - 错误率：高
  - 系统稳定性：中
  - 下游影响：高（执行与复盘口径被污染）
- 问题性质判断：
  - 已验证事实：双击产生两个唯一 action_id。
  - 合理推测：缺少统一幂等键或提交锁闭环。
  - 需进一步验证：高并发下放大程度。
- 复现稳定性：高
- 优化建议：
  - 前端：提交后立即锁定动作按钮至终态返回。
  - 后端：统一强制 `idempotency_key` 去重。
  - 观测：响应返回重复判定字段并在 UI 可见。

### 【问题标题】决策动作未稳定自动承接执行任务，闭环在动作层断开

- 问题级别：高
- 问题类型：业务闭环 / 流程设计
- 发现路径：`/research/decision -> 提交 confirm/reject -> 检查 /api/decision/actions 响应`
- 触发方式：标准路径 + 条件切换
- 表面现象：confirm/reject 响应均未包含 `execution_task`。
- 深层问题判断：动作到执行任务自动承接在当前运行环境未有效生效，闭环承诺与运行结果不一致。
- 影响分析：
  - 用户理解成本：中
  - 任务完成率/转化率：高影响（需人工补链）
  - 操作效率：高影响
  - 信任感：高影响
  - 错误率：中
  - 系统稳定性：中
  - 下游影响：高（复盘链缺失）
- 问题性质判断：
  - 已验证事实：真实响应缺失执行任务字段。
  - 合理推测：版本未生效/触发条件覆盖不足。
  - 需进一步验证：服务更新后是否恢复。
- 复现稳定性：高
- 优化建议：
  - API 合约化：`confirm` 必须返回执行任务标识（或明确失败原因）。
  - 映射固化：`confirm/reject/defer/watch` 全覆盖映射与回写。
  - 页面证据：决策页显式展示“自动生成执行任务ID”。

- 深层问题判断：路由上下文存在但查询层未消费，导致用户误以为看到“该动作订单”。
- 影响分析：
  - 用户理解成本：高
  - 任务完成率/转化率：中高
  - 操作效率：高（人工筛选）
  - 信任感：高
  - 错误率：中高（误读归属）
  - 系统稳定性：中
  - 下游影响：高（审计与复盘链弱化）
- 问题性质判断：
  - 已验证事实：API 请求未包含过滤参数。
  - 合理推测：Orders 页面未消费路由 query。
  - 需进一步验证：后端是否支持该过滤字段。

## 四、隐藏规则与系统机制判断

### 已验证

1. 决策动作返回 `evidence_warnings`，存在证据链软约束机制。  
2. 受限路由升级页包含任务说明、权限说明与恢复路径。

4. 重复提交可生成多条动作，幂等保护不足。  

### 合理推测

1. 动作自动承接执行任务逻辑可能受部署版本或触发条件影响。  
2. Quick Insight 入口可能存在于非对象页。  
3. 候选漏斗与上游导入依赖后台任务，不一定在前台显式可控。

## 五、未完成验证项与风险推断

### 1) 支付/审核/强风控链路

- 阻塞原因：当前站点未暴露该业务流程。
- 已尝试步骤：多角色登录、受限路由、写操作边界试探。
- 已确认事实：权限门禁可解释。
- 风险推断：若后续引入支付/审核，现有跨页上下文不一致问题可能放大。

### 2) 候选漏斗高并发冲突链路

- 阻塞原因：缺前台可控并发造数入口。
- 已尝试步骤：漏斗页访问与操作可达性检查。
- 已确认事实：漏斗页面可访问。
- 风险推断：高并发触发下可能出现状态口径不一致，需接口级压测验证。

### 3) Quick Insight 端到端消费链

- 阻塞原因：当前路径未发现稳定入口，无法完整覆盖“快读->深挖分流”。
- 已确认事实：后端快速接口能力存在。
- 风险推断：能力触达不足导致业务价值释放受限。

## 六、最值得优先处理的问题排序

1. 决策动作重复提交无幂等（高优先）  
   - 影响链路：决策动作闭环  
   - 类型：短期修补 + 中期机制固化  

2. 动作到执行任务自动承接不稳定（高优先）  
   - 影响链路：决策 -> 执行 -> 复盘  
   - 类型：机制修补（非纯前端文案问题）  

3. 订单页深链参数无效（高优先）  
   - 影响链路：决策追踪链  
   - 类型：短期修补（前后端参数对齐）  

4. 评分页决策入口可达性不稳定（中高）  
   - 影响链路：研究入口转化  
   - 类型：短期修补  

5. Quick Insight 入口可发现性不足（中）  
   - 影响链路：快速分析链  
   - 类型：IA 优化 + 交互收口  

6. 权限边界对用户不透明（中）  
   - 影响链路：权限门禁链  
   - 类型：说明增强 + 角色化治理  

若只能先改 3 个：  
1) 重复提交幂等  
2) 动作自动承接执行任务  
3) 订单页深链参数生效  

---

## 七、结论

当前系统的主要问题不是“功能缺失”，而是“功能背后的业务机制不够可靠、闭环不够硬”。  
系统已超出原型阶段，但距离“稳定支撑业务运行”的决策产品仍有差距。后续迭代应优先补三层能力：  

1. 机制层：幂等、自动承接、参数一致性  
2. 闭环层：动作到执行到复盘的可追踪自动化  
3. 入口层：主链路可发现、可预期、可复核
