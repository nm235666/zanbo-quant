# 项目完成态（最终形态）定义文档（2026-04-19）

## 1. 文档目的

本文件定义项目“完成态”目标，不是最小可行版本，而是可长期运行的目标产品形态。  
核心目标：让用户每天进入系统后，可以直接完成**短线执行决策**与**长线宏观应对决策**。

---

## 2. 最终一句话定义

系统应成为“日度可执行的双周期投资操作台”：  

- 短线：今天买什么、卖什么、买卖多少、为什么、何时失效。  
- 长线：宏观短中长期状态是否变化、变化意味着什么、组合层应采取什么动作。  

---

## 3. 定义级约束（硬约束）

### 3.1 时间口径统一（分层 SLA）

- `3 分钟判断 SLA`：系统应支持用户在 3 分钟内完成核心信息判断。  
  具体包含：今日短线主题、重点标的、卖出原因、宏观三周期是否变化。
- `5 分钟动作 SLA`：系统应支持用户在 5 分钟内形成首个可执行动作。  
  具体包含：完成首个买入、卖出、减仓、持有或观察动作确认。

### 3.2 股池三层结构（统一对象模型）

系统中的“股池”固定为三层：

1. 系统机会池：全市场机会发现层（发现什么值得看）。  
2. 用户关注池：研究承接层（决定什么值得重点研究和准备动作）。  
3. 用户持仓池：实际仓位管理层（当前手里的票该怎么管）。

默认主流向：

`系统机会池 -> 用户关注池 -> 用户持仓池`

允许的逆向/退出流转：

- `用户持仓池 -> 用户关注池`（卖出后继续观察）  
- `用户关注池 -> 淘汰`（观察后放弃）  
- `用户持仓池 -> 淘汰`（清仓并退出跟踪）

约束说明：

- “淘汰”表示退出当前决策范围，不等于删除历史记录。  
- 历史流转、动作、执行与复盘轨迹必须保留，可追溯、可回放。

### 3.3 仓位口径统一（账户级比例）

“买多少/卖多少”必须统一为“用户当前整体账户仓位占比”口径输出，而非孤立单票数量。

输出示例（完成态）：

- 某标的建议配置为总仓位的 `5%-8%`  
- 某主题总暴露不超过总仓位 `20%`  
- 某标的减仓至原仓位 `50%`  
- 某类策略仓位整体下调 `X%`

### 3.4 短线与长线冲突裁决（必须自动给结论）

当“短线进攻信号”与“长线防守状态”冲突时，系统必须输出裁决结果，不能留给用户自行消化。

裁决原则：

- 允许局部高确定性短线参与，但同步压缩总风险预算。  
- 同步执行长线防守动作（提高现金、限制总仓位、限制主题集中度、暂停高波动策略）。  
- 在极端防守状态下，允许系统直接建议空仓或极低风险暴露。

### 3.5 长线页面实体（信息架构硬落点）

长线能力必须绑定明确页面实体：

- `/macro/regime`：宏观三周期状态页（短/中/长期状态、变化原因、可信度）。  
- `/portfolio/allocation`：长线配置动作页（风险敞口、行业权重、防守/进攻切换、现金比例）。

### 3.6 双模式前端组织（信息架构约束）

当前系统前端必须按“用户模式 / 后台管理模式”进行双模式组织。

- 用户模式承接研究、判断、决策、执行、复盘主链。
- 后台管理模式承接权限、任务、配置、监控、审计主链。
- 公共共享页层承接登录、升级、错误、回调等跨模式入口与兜底状态。

具体路由、页面归属、共享页层、对象上下文规则与迁移规则，以
[前端双模式重构执行方案](/home/zanbo/zanbotest/docs/frontend_dual_mode_refactor_plan_2026-04-20.md)
为准。

## 4. 完成态用户体验（每日）

## 4.1 登录后 3 分钟内必须回答的 6 个问题

1. 今天最该关注的短线主题是什么（Top N，按优先级排序）  
2. 用户关注池里哪些标的应“新开仓”  
3. 每只新开仓标的建议仓位是多少  
4. 用户持仓池里哪些标的应“减仓/清仓”  
5. 卖出原因是什么（止盈/止损/失效/风险切换）  
6. 宏观短期/中期/长期状态是否变化，组合层该怎么应对  

## 4.2 当日工作流（完成态）

1. 进入 `研究工作台`（唯一主入口）  
2. 查看 `今日短线主题榜` + `主题冲突裁决`  
3. 查看 `用户关注池开仓建议 + 用户持仓池调整建议` 动作卡（买/卖/持有/观察）  
4. 一键确认动作进入 `执行队列`（自动关联仓位、风控、失效条件）  
5. 在 `执行看板` 跟踪状态（待执行/执行中/已执行/取消）  
6. 收盘后自动进入 `复盘看板`（执行偏差、结论有效性、规则修正建议）  
7. 同时查看 `宏观三周期状态卡` 与组合动作建议（防守/进攻/轮动）

## 5. 产品信息架构（完成态）

本节关于研究链、治理链和默认入口的长期边界，同时受
[前端双模式重构执行方案](/home/zanbo/zanbotest/docs/frontend_dual_mode_refactor_plan_2026-04-20.md)
约束。

## 5.1 一级结构（研究用户视角）

1. 今日工作台（唯一主入口）  
2. 短线执行  
3. 长线配置  
4. 复盘与追踪  
5. 系统与治理（仅管理员）

## 5.2 核心页面（必须长期稳定）

- `/research/workbench`：今日总入口（任务优先队列 + 主题结论 + 动作建议）
- `/market/conclusion`：市场结论页（冲突裁决后单口径输出）
- `/research/funnel`：候选漏斗（候选进入、晋级、淘汰、卡点）
- `/research/decision`：决策板（动作确认、证据链、可解释结论）
- `/portfolio/orders`：执行单与状态
- `/portfolio/positions`：当前持仓与仓位结构
- `/portfolio/review`：复盘与策略修正
- `/macro/regime`：宏观三周期状态页（状态变化与可信度）
- `/portfolio/allocation`：长线配置动作页（组合层应对动作）

## 6. 短线投资完成态能力

## 6.1 主题层（今日该关心什么）

- 输出“今日短线主题 Top N”  
- `N` 默认值为 `Top 3`，并支持按用户配置扩展为 `Top 5 / Top 10`。  
- 排序依据包括：  
  - 时效性  
  - 信号强度  
  - 资金确认度  
  - 冲突风险  
- 必须有“冲突裁决口径”：多源冲突时统一给单一结论，不让用户自己拼。

## 6.2 标的层（买什么/卖什么）

- 对每只标的输出标准化动作卡：  
  - `动作类型`：新买/加仓/减仓/卖出/观察  
  - `建议仓位`：目标仓位区间  
  - `触发原因`：主题、信号、事件、风险  
  - `失效条件`：价格、时间、事件、宏观状态变化  
  - `优先级`：高/中/低  

## 6.3 仓位层（买卖多少）

- 系统必须具备仓位建议引擎：  
  - 单票仓位上限  
  - 主题集中度上限  
  - 日内总风险预算  
  - 回撤保护规则  
- 短线仓位建议必须以“账户整体仓位占比”输出。  
- 给出“仓位变化建议”，不是只给方向建议。

## 6.4 执行层（动作变成执行）

- 决策动作自动承接执行任务（无手工补链）  
- 状态可追踪：planned -> executing -> done/cancelled  
- 跨页追踪一致：action_id、order_id、execution_status 全链可回查。

---

## 7. 长线投资完成态能力

## 7.1 宏观三周期状态机

- 短期（1-4 周）  
- 中期（1-6 个月）  
- 长期（6-24 个月）  

每个周期必须输出：

- 当前状态（扩张/放缓/风险抬升/震荡等）  
- 相比上一周期是否变化  
- 变化原因证据  
- 可信度评分

## 7.2 宏观变化到组合动作映射

每个周期状态变化必须自动映射到组合级动作建议：

- 增减风险敞口  
- 行业权重调整  
- 防守资产/进攻资产切换  
- 现金比例建议  
- 暂停/恢复某类短线策略

## 7.3 短长线冲突裁决（执行约束）

- 当短线建议与长线状态冲突时，系统必须给出统一裁决动作。  
- 输出必须包含：  
  - 允许执行的局部短线动作范围  
  - 必须同步执行的长线防守动作  
  - 风险预算压缩比例与生效条件

## 7.4 长线复盘

- 记录“宏观判断 -> 组合动作 -> 后续结果”  
- 评估判断有效性与动作有效性，形成策略修正建议。

---

## 8. 决策闭环（必须成立）

完整闭环定义：

`输入 -> 分析 -> 判断 -> 动作 -> 执行 -> 跟踪 -> 反馈 -> 复盘 -> 规则修正`

完成态要求：

- 任何动作都能追溯其输入证据。  
- 任何执行都能追溯其来源动作。  
- 任何复盘都能反向定位原始判断。  
- 任何规则修正都能说明由哪些复盘结果触发。  

---

## 9. 数据与规则治理完成态

## 9.1 规则可见化

- 用户能看到：  
  - 为什么给出该结论  
  - 为什么给出该仓位  
  - 为什么触发卖出/暂缓  
  - 当前受哪些风控约束  

## 9.2 状态管理要求

- 幂等：重复提交不产生重复动作。  
- 一致性：跨入口、跨页面结果一致。  
- 恢复性：中断后能恢复上下文与任务状态。  
- 延迟可见：异步任务有明确状态与下一步指引。  

## 9.3 权限策略要求

- 研究用户只见研究链，不被运维功能干扰。  
- 升级页明确“受限任务、缺失权限、替代路径”。

---

## 10. 验收标准（完成态）

## 10.1 业务结果指标

- 用户在 `3 分钟` 内完成核心信息判断（主题、重点标的、卖出原因、宏观变化）  
- 每日首次登录到形成首个可执行动作 <= 5 分钟  
- “建议动作 -> 执行任务”自动承接率 = 100%  
- 执行任务可追溯覆盖率 = 100%  
- 复盘回填率 >= 90%  
- 短线主题结论页日访问覆盖率 >= 80%（研究角色）
- 宏观三周期状态页或长线配置动作页日访问覆盖率 >= 60%（研究角色）

## 10.2 产品与流程指标

- 主入口使用率（workbench）>= 85%  
- 单票研究跨页跳转次数中位数 <= 2  
- 重复提交导致重复动作率 <= 0.5%  
- 权限受限场景下“有解释+有替代路径”覆盖率 = 100%

## 10.3 工程与回归指标

- 主链 e2e 全通过（研究入口、决策、执行、复盘、升级页）  
- 核心写操作具备幂等校验与错误码可行动提示  
- 指标门禁可阻断不达标发布。

---

## 11. 阶段性边界（防止偏航）

不属于本阶段完成态必需项：

- 实盘券商直连下单（可后续扩展）
- 花哨可视化优先于决策闭环
- 与主链无关的大规模重构

优先顺序固定：

1. 闭环可靠性  
2. 决策可执行性  
3. 入口收口与信息清晰  
4. 视觉与表达升级

---

## 12. 完成态判定

当且仅当以下条件同时满足，项目才视为达到目标形态：

1. 用户每天登录后无需跨多个模块拼装，就能基于系统机会池、用户关注池、用户持仓池，得到“买什么/卖什么/仓位多少/主题优先级”。  
2. 短线动作可自动承接执行并可复盘。  
3. 长线宏观三周期变化可转化为可执行的组合动作。  
4. 全链路可追踪、可解释、可回放。  
5. 回归门禁能稳定保障上述能力不回退。
