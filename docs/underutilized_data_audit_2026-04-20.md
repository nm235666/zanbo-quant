# 当前项目未充分利用数据审计（2026-04-20）

## 1. 文档目的

这份文档不是盘点“系统里有没有数据”，而是审计“系统里已经存在的数据，有多少真正进入了业务主链，并转成了可执行的研究、决策、执行与复盘能力”。

判断标准直接对齐完成态目标：

- 是否能帮助用户在 `3 分钟` 内完成核心判断
- 是否能帮助用户在 `5 分钟` 内形成首个可执行动作
- 是否能支撑三层股池流转
- 是否能支撑账户级仓位建议
- 是否能支撑短长线冲突裁决
- 是否能打通 `输入 -> 分析 -> 判断 -> 动作 -> 执行 -> 跟踪 -> 反馈 -> 复盘 -> 修正`

因此，本文件不是纯数据库盘点文，而是“完成态还缺什么数据能力”的主链审计文。

---

## 2. 审计方法

### 2.1 审计分类

本次把“未充分利用数据”分成两类：

1. `已有但基本没吃进去`
- 数据真实存在，后端或数据库中已沉淀
- 但前端主链几乎看不到，或只在详情/后台中零散出现

2. `已经接入但只停留在展示/弱提示，没有形成动作`
- 数据已经进入页面、评分、摘要或说明
- 但还没有转成候选筛选、动作卡、仓位建议、风险裁决或复盘修正规则

### 2.2 统一判断维度

每类数据统一按以下维度判断：

- 当前数据源
- 当前在系统中的使用位置
- 未利用点
- 若充分利用，可转成什么能力
- 优先级
- 主要落点页面

## 5. 优先级排序

## 5.1 P0：立刻能提升“今日买卖与仓位建议”

1. 资金流数据：`capital_flow_market / capital_flow_stock`
2. 风险情景：`risk_scenarios`
3. 公司事件：`stock_events`
4. 日报总结：`news_daily_summaries`

优先原因：

- 这四类最容易直接进入工作台、市场结论、决策板和持仓页
- 能最快提升“今天买什么、卖什么、仓位多少、为什么”的表达质量

## 5.2 P1：提升候选/信号可信度与主线解释力

1. 群聊元数据：`chatroom_list_items`
2. 信号状态机：`signal_state_tracker / signal_state_events`
3. 主题映射与别名归一：`theme_stock_mapping / stock_alias_dictionary`

优先原因：

- 这三类更像过滤器、可信度修正器和关系放大器
- 能显著减少“有候选但不知该不该信”的问题

## 5.3 P2：提升长线质量约束与复盘学习能力

1. 公司治理：`company_governance`
2. 宏观原始序列自动生成三周期状态：`macro_series / fx_daily / rate_curve_points / spread_daily`
3. 复盘反哺规则：执行、结果、风控与来源有效性的闭环回写

优先原因：

- 这类更偏长期骨架增强
- 对“长期可运行的双周期投资操作台”非常重要，但短期感知没有 P0 那么直接

## 6. 下一步拆分建议

## 6.1 适合先做前端表达升级的项

- 资金流与风险情景进入动作卡、持仓卡、市场结论卡
- 日报总结进入市场结论页主摘要
- 信号状态机在研究侧页面做口语化翻译
- 群聊来源质量分进入候选与信号的说明层

特点：

- 不一定先改底层协议
- 可以先通过已有接口聚合结果，把价值表达出来

## 6.2 需要先补后端聚合/规则层的项

- 资金流确认度统一口径
- 风险情景到仓位/失效条件的规则映射
- 事件驱动的时间窗口规则
- 主题集中度与组合重复暴露计算
- 群聊元数据对候选/信号的权重修正

特点：

- 这类若没有统一后端口径，前端会变成“多处各算一遍”
- 应优先通过聚合层或决策层做统一计算

## 6.3 需要先建立验收口径再做实现的项

- 宏观原始序列自动生成三周期状态
- 复盘结果反向修正规则
- 哪类数据应成为卖出主因、哪类数据只做弱提示
- 群聊与新闻、信号、评分之间的最终优先级关系

建议验收口径：

- 是否缩短到 3 分钟判断 SLA
- 是否提升 5 分钟首个动作形成率
- 是否减少候选误入池、错误晋级和无依据动作
- 是否提升动作可追溯性和复盘可解释性

---

## 7. 总结

当前项目真正的问题不是“缺少数据”，而是“已经有不少高价值数据，但还没有被压缩成用户每天能直接消费的决策能力”。

最核心的缺口不是数据库覆盖率，而是数据价值转译率：

- 是否能从素材变成结论
- 是否能从结论变成动作
- 是否能从动作变成执行与复盘

后续产品化和验收的重点，不应只是继续加页面，而应优先提升这些现有数据进入主链后的转化效率。
